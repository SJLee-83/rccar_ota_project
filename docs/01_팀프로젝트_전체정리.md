# SDV 시대의 RC카 — 무선 OTA 업데이트와 AI 음성 제어

> SSAFY 임베디드 트랙 15기 · 1학기 관통 PJT
> 팀 구성: 박찬혁, 이승재(마루)

---

## 1. 프로젝트 개요

차량은 출고 후에도 소프트웨어 업데이트로 기능이 확장되는 SDV(Software Defined Vehicle) 구조로 전환되고 있다. 본 프로젝트는 이 개념을 RC카 규모로 구현한 것으로, 두 개의 핵심 기능을 통합했다.

- **기능 A — 무선 OTA 업데이트 및 주행 제어**: 제어 MCU에 무선 모듈이 없어도 게이트웨이(ESP32)를 통해 펌웨어를 무선으로 배포하고, 바뀐 펌웨어가 실제 주행 동작 변화로 이어지는 것을 시연.
- **기능 B — AI 음성 제어**: 사용자의 음성을 차량 명령으로 변환하는 인포테인먼트 기능.

핵심 메시지는 "현장에서 보드를 분해하거나 재배선하지 않고, 무선 명령만으로 차량 기능을 안전하게 바꾼다"이다.

## 2. 시스템 아키텍처

명령과 상태 피드백이 MQTT / SPI / I2C·GPIO 경로로 이어지는 구조다.

```
[PC 관제 GUI]                    [Raspberry Pi 5]
 PySide6                          SPH0645 I2S 마이크
   │ MQTT                           │
   │ (OTA/Command, OTA/Data,        │ STT→LLM→TTS (GMS Proxy)
   │  OTA/Status, RC 명령)          │ 음성→차량 명령 변환
   ▼                                ▼
┌─────────────────────────────────────────┐
│           MQTT Broker (Mosquitto, 1883)  │
└─────────────────────────────────────────┘
   │ MQTT
   ▼
[ESP32 게이트웨이]
   │ SPI (Master) + BUSY GPIO 핸드셰이크
   │ OTAS / metadata / OTAD / OTAE 프레임
   ▼
[Renesas RA6E1 (SPI Slave)]
   │ Dual-Bank Code Flash 제어
   │ CRC32 검증 → BankSwap → Reset
   │ I2C
   ▼
[PCA9685 Motor HAT] → 서보(조향) + DC 모터(구동)
   +
[HC-SR04 초음파 센서] → 전방 충돌 감지
```

### 통신 경로 요약

| 구간 | 프로토콜 | 용도 |
|------|----------|------|
| GUI ↔ Broker ↔ ESP32 | MQTT | OTA 데이터, RC 조종 명령, 상태 피드백 |
| ESP32 ↔ RA6E1 | SPI + BUSY GPIO | 펌웨어 청크 전달, 명령, 핸드셰이크 |
| RA6E1 ↔ Motor HAT | I2C | 서보·DC 모터 제어 |
| RA6E1 ↔ HC-SR04 | GPIO (TRIG/ECHO) | 전방 거리 측정 |
| Pi5 ↔ GMS | HTTPS | STT / LLM / TTS API |

## 3. 기능 A — OTA 업데이트 (Dual-Bank 방식, 최종 채택)

### 3.1 왜 무선 업데이트인가

- **BEFORE**: 버전을 바꿀 때마다 보드 분해 → J-Link 케이블 직접 연결. 현장 차량은 분해 자체가 어렵고, 잘못 쓰면 보드가 벽돌(Bricked) 상태가 됨.
- **AFTER**: PC에서 `.bin` 선택 한 번으로 무선 배포. 재배선 없이 기능 교체.

### 3.2 Dual-Bank 구조 — 부트로더 없이 갱신

RA6E1의 1 MiB Code Flash를 512 KiB × 2 뱅크로 운용한다. 별도 부트로더 영역이나 애플리케이션 점프 코드를 만들지 않고, RA6E1 하드웨어가 제공하는 Dual-Bank와 시작 뱅크 선택 기능을 사용한다.

| 논리 주소 | 역할 |
|-----------|------|
| `0x00000000` | 현재 선택된 Active Bank가 매핑되어 CPU가 실행 |
| `0x00200000` | 반대쪽 Inactive Bank 접근용 FSP Dual-Bank alias |

핵심 원리: **실행 중인 Active Bank 애플리케이션 자신이 다음 펌웨어를 받아 반대쪽 Inactive Bank에 기록**한다. Bank Swap은 CPU가 다른 주소로 점프하는 게 아니라, 다음 Reset부터 어느 물리 뱅크가 `0x00000000`에 매핑될지를 바꾸는 동작이다. 따라서 새 `.bin`도 `0x00000000` 기준으로 링크한다.

이 구조 덕분에 다운로드나 검증이 실패해도 Bank Swap만 하지 않으면 기존 Active Bank로 계속 동작한다.

### 3.3 OTA 실행 순서

1. GUI가 펌웨어 크기·청크 수·전체 CRC32를 담은 `OTA_START` 전송
2. ESP32가 SPI로 `OTAS` + 16바이트 metadata 전달
3. RA6E1이 metadata CRC와 이미지 범위 검사
4. RA6E1이 `0x00200000`의 필요한 Flash 블록만 Erase
5. GUI가 256바이트 청크를 순차 전송
6. ESP32가 각 청크를 `OTAD` SPI 프레임으로 변환
7. RA6E1이 청크 순서·CRC 확인 후 Inactive Bank에 기록
8. 기록 성공한 청크에만 ACK 반환 (실패 시 NACK → 재전송)
9. 모든 청크 완료 후 GUI가 `OTAE` 전송
10. RA6E1이 크기·청크 수·전체 Running CRC 최종 확인
11. 모든 검증 성공 시에만 `R_FLASH_HP_BankSwap()` 호출
12. `NVIC_SystemReset()`으로 Reset → 새 뱅크가 `0x00000000`에 매핑 → 새 펌웨어 실행

### 3.4 안정성 강화 요소

개발 과정에서 발견·해결한 문제와 대응:

- **SPI 동기화 오작동**: RA6E1이 전송 완료를 기다리지 않고 버퍼를 반복 읽어, 펌웨어 바이너리 속 `w/a/s/d` 바이트가 RC 명령으로 오인되어 모터가 임의 동작. → `spi_callback()`에서 `SPI_EVENT_TRANSFER_COMPLETE` 확인 후 진행하도록 blocking 처리. OTA 모드 중 일반 RC 명령 차단.
- **초기화 데드락**: BUSY 핀(P302) 초기값을 LOW로 명시하지 않아 ESP32가 무한 대기. → `hal_entry()` 최상단에서 `SET_BUSY_LOW()` 명시.
- **GUI 크로스스레드 크래시**: MQTT 콜백(백그라운드 스레드)에서 UI 직접 조작으로 충돌. → Signal & Slot 패턴으로 메인 스레드에 위임.
- **데이터 무결성**: 청크 1개만 깨져도 벽돌화. → CRC32(metadata/청크/전체) + ARQ 재전송(NACK 0x1F, 최대 20회) + Session ID + 바이너리 프로토콜.

### 3.5 이 방식의 한계

새 이미지가 정상 기록됐는지는 검증하지만, 실제로 부팅하여 정상 동작하는지 확인 후 자동 롤백하는 기능은 없다. 새 뱅크가 부팅되지 않으면 e2 studio 디버거로 정상 펌웨어를 직접 기록해야 한다. (향후: 첫 부팅 성공 플래그, Heartbeat 롤백, 서명 검증 등으로 보완 가능.)

## 4. 기능 A 부속 — HC-SR04 충돌 회피 (OTA 가치 시연)

동일한 RC카에 두 펌웨어를 OTA로 번갈아 올려 기능 변화를 직접 보여주는 시연용 구성.

- **V1.0.0 (업데이트 전)**: HC-SR04 핀 초기화 없음. 전진 중 장애물이 있어도 계속 주행 (사람이 직접 멈춰야 함).
- **V2.0.0 (업데이트 후)**: 전방 20cm 이내 장애물 감지 시 즉시 정지. 장애물 제거를 28cm 이상에서 3회 연속 확인해야 전진 재허용(hysteresis). 후진은 탈출을 위해 항상 허용.

| 항목 | 값 |
|------|-----|
| TRIG 핀 | P409 / Arduino D2 |
| ECHO 핀 | P104 / Arduino D3 (5V→3.3V 분압 필수) |
| 정지 거리 | 20cm |
| 해제 거리 | 28cm × 3회 연속 |
| 측정 주기 | 약 60ms, 비차단 상태 머신 |
| 거리 공식 | ECHO HIGH 시간(us) / 58, DWT 사이클 카운터 사용 |

기존 SPI(P100~P102), I2C(P400/P401), BUSY(P302)와 겹치지 않는 핀을 선정했다.

## 5. 기능 B — AI 음성 제어

라즈베리파이 5 + SPH0645 I2S 마이크로 음성을 캡처하고, SSAFY GMS 프록시를 통해 STT → LLM → TTS 파이프라인을 실행하는 Qt GUI.

```
SPH0645 → WAV 녹음 → Whisper STT → LLM → TTS → 스피커 재생
```

- 음성 질문: 기본 5초 녹음 후 전체 과정 자동 실행
- 텍스트 질문: 마이크 없이 LLM·TTS 구간만 시험 가능
- API 키는 소스가 아닌 `.env`에서만 로드
- 원류: 이전에 진행한 "음성인식 센서 + 라즈베리파이4 음성 RC카 제어 실습"을 ESP32+RA6E1 구성에 맞게 수정·적용

배선 요점: SPH0645는 3.3V 장치(5V 금지). BCLK=GPIO18, LRCLK=GPIO19, DOUT=GPIO20, SEL=GND(Left). Pi5는 아날로그 오디오 단자가 없어 USB/HDMI/BT 스피커 별도 필요.

## 7. OTA 두 방식의 병행 연구

이 프로젝트의 OTA는 두 접근을 병행 시도한 뒤 하나를 최종 채택한 결과다. Dual-Bank 방식은 완성되어 시연에 사용되었고, MCUboot 방식은 슬롯 교체 검증까지 도달했으나 **무선 전송 부분(self-programming)을 완성하지 못해 미완으로 남았다.**

| 항목 | Dual-Bank (박찬혁, 채택) | MCUboot (이승재, 연구) |
|------|------------------------|----------------------|
| 부트로더 | 없음 (앱이 직접 수행) | 있음 (MCUboot) |
| 메모리 | Bank A/B 512KiB, alias 0x00200000 | Primary/Secondary (0x10000/0x88000) |
| 펌웨어 교체 | `R_FLASH_HP_BankSwap()` + Reset | 부트로더 Overwrite 복사 |
| 검증 | CRC32 (metadata/청크/전체) | ECDSA P-256 서명 |
| self-programming 제약 | 하드웨어 Dual-Bank로 우회 | 직면 (실행 중 코드플래시 쓰기 hang) |
| 최종 시연 | ✅ 채택 (완성) | 슬롯 교체 검증까지 (미완성) |

두 방식 모두 배운 점과 고려 과정이 있어, 마루의 MCUboot 방식은 별도 저장소에 독립 정리한다.

**→ MCUboot 방식 상세: [rccar-mcuboot-ota 저장소] (링크 예정)**

## 8. 한계 및 개선 여지 (회고)

다음 프로젝트 주제 선정을 위한 회고 재료로, 이번 프로젝트에서 드러난 부족한 점과 발전 방향을 한곳에 정리한다. 기술적 미완뿐 아니라 개발 프로세스 차원의 공백도 포함한다.

### 8.1 OTA 안전성 — 자동 롤백 부재 (가장 큰 공백)

두 방식(Dual-Bank, MCUboot) 공통의 한계다. 새 펌웨어가 정상 기록됐는지는 검증하지만, **실제로 부팅하여 정상 동작하는지 확인 후 실패 시 이전 버전으로 자동 복구하는 기능이 없다.** 새 뱅크/슬롯이 부팅에 실패하면 e2 studio 디버거로 직접 펌웨어를 다시 기록해야 한다 — 즉 "무선"의 이점이 사라지고 현장 복구가 불가능해진다.

실무 SDV에서 가장 중요한 부분이며, 발전축으로:
- 새 펌웨어 첫 부팅 성공 플래그 (confirm/revert)
- 일정 시간 내 Heartbeat 없으면 이전 뱅크로 자동 롤백
- MCUboot의 경우 swap 모드(Overwrite 대신)로 롤백 지원

### 8.2 보안 — 무선 완성과 서명 검증의 트레이드오프

두 방식이 서로 다른 절반씩만 달성했다.

| | 무선 전송 완성 | 서명 검증 |
|---|:---:|:---:|
| 팀 Dual-Bank (채택) | ✅ | ❌ (CRC32만) |
| 마루 MCUboot | ❌ (self-programming 벽) | ✅ (ECDSA P-256) |

채택된 Dual-Bank는 CRC32로 **전송 오류**는 잡지만 **악의적 변조**는 막지 못한다. 서명 없는 무선 OTA는 임의 펌웨어 주입에 취약하다. → **"무선 완성 + 서명 검증"을 모두 갖춘 보안 OTA**가 자연스러운 다음 단계이며, 두 방식의 장점을 결합하는 방향이 된다.

### 8.3 AI 음성과 차량 제어의 통합도

ai_audio는 STT→LLM→TTS 파이프라인은 동작하지만, 음성 인식 결과가 **실제 차량 제어 명령으로 얼마나 폭넓게·안정적으로 연결되는지**는 제한적이다. 인포테인먼트 응답과 실제 주행 명령 사이의 연결이 얕다. → 음성/비전 기반 명령 확장, 자율 주행 요소 결합 여지.

### 8.4 개발 프로세스 성숙도

- **버전 관리 부재**: 프로젝트 진행 중 git을 사용하지 않아 개발 타임라인·시행착오·기여 이력이 커밋으로 남지 않았다. 결과물을 사후에 일괄 업로드한 상태.
- **통합 테스트 체계 부재**: 통합 검증이 수동 "반복"에 의존. 자동화된 테스트나 회귀 검증 없음.
- **개선 방향**: 처음부터 git 브랜치 전략 적용, CI 도입, HIL(Hardware-in-the-Loop) 테스트 등 개발 프로세스 자체를 다음 프로젝트의 서브 목표로 삼을 수 있다.

### 8.5 self-programming 미해결 (개인 기술 부채)

마루의 MCUboot 방식에서 앱이 실행 중 코드플래시에 직접 쓰는 부분(C-7b)이 RAM 실행 이슈로 미완이다. 상세는 MCUboot 정리 문서 참조. 임베디드 심화 역량으로 이어지는 학습 목표.
