# 🛠️ OTA 최근 트러블슈팅 및 버그 픽스 요약

이 문서는 최근 발생했던 GUI 크래시 문제와 RA6E1 보드의 OTA/Ping 먹통 현상을 해결하기 위해 분석하고 적용한 핵심 해결책을 요약한 문서입니다. 기존 메인 개발 로그를 보존하기 위해 별도의 문서로 분리하여 작성되었습니다.

---

## 1. GUI 크래시 현상 (Cross-thread UI Update Issue)
### 🚨 증상
- PyQt(PySide6) 기반의 PC 클라이언트에서 펌웨어 업로드를 진행하는 도중 프로그램이 강제 종료(팅김)되는 현상 발생.
- 에러 메시지: `Shiboken::Conversions::_pythonToCppCopy: Cannot copy-convert ... to C++.`

### 🔍 원인
- MQTT 수신 콜백 함수(`on_message`)는 백그라운드 네트워크 스레드에서 실행되는데, 이 스레드에서 직접 GUI 컴포넌트(예: `ui.progress_bar.setValue()`, `ui.status_log.append()`)를 제어하려고 시도함.
- Qt 프레임워크에서는 메인(GUI) 스레드가 아닌 다른 스레드에서 UI를 직접 조작하는 것을 엄격하게 금지하므로 충돌이 발생함.

### ✅ 해결책
- **Signal & Slot 패턴 적용**: 백그라운드 스레드에서 직접 UI를 변경하지 않고, 커스텀 시그널(`sig_append_log`, `sig_chunk_ack`, `sig_chunk_error` 등)을 메인 스레드로 emit(발신)하도록 구조를 변경.
- 메인 스레드에서 시그널을 받아 UI를 안전하게 업데이트하여 크래시 문제를 완벽히 해결함.

## 2. 통신 데드락 (Ping 실패 및 ESP32 Timeout)
### 🚨 증상
- Ping 버튼을 눌러도 RA6E1의 펌웨어 버전이 응답하지 않음 (`Ping Failed! Got=0x00`).
- `Start OTA Update`를 누르면 `▶ Starting OTA Update...` 메시지 이후로 아무런 진행이 되지 않으며 시리얼 모니터에 청크 관련 메시지가 아예 출력되지 않음.
- 보드의 리셋 버튼을 눌러도 반응이 없음.

### 🔍 원인
- **보드 먹통 (Bricked) 상태 유지**: 이전 OTA 테스트 중 펌웨어가 파괴된 상태 그대로 남아있었기 때문에, RA6E1의 SPI 통신 드라이버 자체가 동작하지 않아 `0x00`만 반환됨.
- **초기화 데드락 버그**: RA6E1의 `hal_entry.c`에서 `PIN_BUSY` (P302) 핀의 초기 상태를 명시적으로 `LOW`로 잡아주지 않음. 부팅 시 핀이 Floating(HIGH) 상태가 되면, ESP32는 통신을 시작하기 위해 `wait_for_busy_low()`에서 무한히 대기하다가 5초 뒤 타임아웃으로 통신을 조용히 취소함.
- **무한 루프 취약점**: RA6E1에서 SPI 통신(`R_SPI_WriteRead`)이 에러를 반환했을 때 예외 처리 없이 `while (!spi_transfer_complete)` 루프에 진입하여 영원히 갇히는 문제 발견.

### ✅ 해결책
- **초기 데드락 방지**: `hal_entry()` 최상단에 `SET_BUSY_LOW();`를 명시적으로 추가하여 ESP32의 오해와 대기를 방지.
- **SPI 예외 처리**: `FSP_SUCCESS`일 때만 무한 루프 대기열에 진입하도록 방어 코드 추가.
- **유선 복구 필수 안내**: 이미 펌웨어가 파괴된 보드는 무선(OTA) 수신이 불가능하므로, J-Link 등 유선 프로그래머를 사용하여 견고해진 최신 `hal_entry.c` 코드를 1회 직접 플래싱하여 보드를 복구하도록 안내함.

## 3. OTA 안정성 및 ARQ 강화
### 🚨 이슈 및 배경
- 단 하나의 청크라도 통신 노이즈로 인해 깨지면 보드가 복구 불능 상태가 됨.
- 기존의 단순 반복이나 에러 무시 로직으로는 무선 환경의 변수를 통제하기 어려움.

### ✅ 해결책
- **ARQ (자동 재전송 요구) 로직**: 청크 무결성 검증 후 에러 시 명시적 NACK(`0x1F`)를 반환하고, GUI에서 해당 청크부터 다시 전송하도록 핑퐁 로직 구현.
- **연속 실패 횟수 상향**: 동일 청크에 대해 일시적인 노이즈를 대비하여 최대 20회까지 재전송을 허용하도록 GUI 수정.
- (※ 유저께서 직접 추가적으로 Binary 프로토콜 전환 및 CRC32 검증, Session ID, GUI Timer 등의 엔터프라이즈급 안정화 로직을 성공적으로 반영하심)
