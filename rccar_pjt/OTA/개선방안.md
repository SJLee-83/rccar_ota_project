# OTA 업데이트 문제 원인 분석 및 해결 방안

## 1. 문제 원인 분석 (Why it failed)

사용자님께서 `개선점.md`를 통해 정확히 짚어주신 문제들의 기술적 원인입니다.

### 1) RC카가 지멋대로 동작하는 문제 (오작동 원인)
* **비동기 타이밍 문제:** ESP32(Master)는 SPI 클럭 신호에 데이터를 실어 보내지만, 르네사스 RA6E1(Slave) 보드는 기존 코드에서 마스터의 클럭 신호를 기다리지 않고 자신의 루프 속도대로 버퍼를 계속 읽었습니다.
* **결과:** 마스터가 한 바이트를 보내는 동안 RA6E1은 이전 바이트를 수십 번 반복해서 읽게 됩니다. 따라서 `O, T, A, S` 시작 시그널이 정상적으로 감지되지 못했고, RA6E1은 OTA 모드로 진입하지 못한 채 수신되는 펌웨어 바이너리(.bin) 데이터를 일반 조종 명령(`w`, `a`, `s`, `d` 등)으로 잘못 해석하여 모터를 임의로 구동시킨 것입니다.

### 2) 데이터 누락 및 클럭 동기화 실패 문제
* **플래시 메모리 쓰기 병목:** ESP32는 매우 빠른 속도로 연속해서 데이터를 쏟아내지만, RA6E1의 플래시 메모리 섹터 쓰기(Write/Erase) 작업은 시간이 꽤 오래 걸립니다.
* **결과:** 플래시 메모리에 쓰는 동안 ESP32가 보내는 클럭과 데이터는 무시되거나 유실(Overrun Error)될 수밖에 없었고, 결국 펌웨어 파일의 중간이 텅 비어버리거나 데이터가 밀리는 누락 현상이 발생했습니다.

## 2. 해결 방안 (How to fix)

### 개선방안 A: 클럭 동기화 및 인터럽트 플래그 대기 (방금 적용 완료)
* **해결책:** 앞서 적용한 패치와 같이, RA6E1(`hal_entry.c`)에서 `R_SPI_WriteRead` 호출 후 즉시 다음 코드로 넘어가는 것이 아니라, **마스터(ESP32)가 클럭을 쏴서 1바이트 수신이 완전히 끝났음을 알리는 콜백 인터럽트 플래그(`spi_transfer_complete`)를 무조건 기다리도록(Blocking) 변경**했습니다.
* **효과:** 데이터 중복 읽기가 차단되고 마스터와 1:1 클럭 동기화가 이루어져 무작위 RC카 오작동이 완벽히 해결됩니다.

### 개선방안 B: 데이터 무결성 검증 (Integrity Check) 로직 추가 (추가 개발 필요)
* 사용자님의 3번 지적처럼, 수신된 펌웨어의 무결성을 검증하는 것이 필수적입니다. 데이터가 중간에 하나라도 끊기면 치명적인 오류(벽돌 현상)가 발생할 수 있습니다.
* **해결책 (파일 크기 검증 로직):**
  1. **[파이썬 GUI]:** 펌웨어의 총 파일 크기(Byte)를 MQTT `OTA_START`와 함께 보냅니다. (현재 이미 구현되어 있음)
  2. **[ESP32]:** `OTA_START`를 받으면 RA6E1에게 `OTAS` 헤더를 보낸 직후, 펌웨어의 **총 파일 크기(Size)**를 4바이트로 쪼개어 전달합니다.
  3. **[RA6E1]:** `OTAS` + `파일 크기(4바이트)`를 받아 예상되는 총 바이트 수를 메모리에 저장해 둡니다. 이후 데이터를 받을 때마다 `received_count`를 1씩 증가시킵니다.
  4. **[무결성 검증]:** ESP32가 `OTAE`(종료) 시그널을 보내면, RA6E1은 `received_count == expected_size` 인지 엄격하게 비교합니다.
  5. **[안전 조치]:** 정확히 일치할 때만 Bank 스왑 및 재부팅을 수행하고, 단 1바이트라도 누락되었다면 업데이트를 전면 취소(Abort)하여 보드의 안전을 지킵니다.
